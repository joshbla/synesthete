# 2026-03-07

### Goal

Get visible, human-confirmable audio-video correspondence from the model. Not novel visualizations yet — just proof that the model is using audio to drive what it generates. Every run should produce output where a person can look at the video, listen to the audio, and say "yes, those are connected."

### Context

The project had stalled. Multiple iterations of architecture, data diversity, and training configs had produced only visual fuzz — the model was generating plausible-looking visualizer output but ignoring the audio conditioning entirely. Previous work had identified this failure mode (documented in `docs/audio_responsiveness_postmortem.md`) and built tooling to address it, but the default training path still had the key anti-shortcut mechanisms turned off.

### What we discovered

1. **The default config leaves audio enforcement disabled.** `audio_shuffle_loss_weight` is 0.0, `match_enabled` is false, and `audio_silence_mix_prob` is 0.0 in `config/default.yaml`. The model has no training incentive to use audio.

2. **The bootstrap config already has the right settings.** `config/bootstrap_audio_debug.yaml` enables shuffle pressure (weight 2.0), uses debug-only visualizers, mixes in silence/dropout audio, disables style randomness, and cranks audio guidance at inference. This was the config that should have been the starting point all along.

3. **Literature comparison confirmed the approach.** Papers like ASVA/AVSyncD, TPoS, and TempoToken all achieve audio-video synchronization by starting from constrained, auditable tasks with pretrained priors — not by training from scratch on diverse data at small scale. Our bootstrap approach (narrow task, strong enforcement, prove correspondence first) is aligned with how successful projects are structured.

4. **3 epochs of diffusion training was not enough.** After 3 epochs the audio reactivity eval showed only 1.5% pixel difference between correct and shuffled audio — invisible to the eye. The loss was still actively decreasing.

5. **33 total epochs produced measurable correspondence.** After resuming training for 30 more epochs (loss dropped from 0.65 to 0.20), the audio reactivity eval showed 6.9% pixel-space difference — a 4.6x improvement. The model learned to go dark during silence.

6. **Previous-latent conditioning introduces visible delay.** With `prev_latent_weight: 0.5`, audio-driven changes (like darkening during silence) were noticeably delayed by several frames. Setting it to 0 at inference reduced the lag.

### What we changed

- **`src/train_diffusion.py`**: Added checkpoint loading in `DiffusionTrainer.__init__` so `--force-diffusion` resumes from existing weights instead of reinitializing from scratch.
- **`config/bootstrap_audio_debug.yaml`**: Increased diffusion epochs from 3 to 30.

### Current state

The model has learned the coarsest audio correspondence: silence → dark, sound → not dark. The response is still somewhat delayed and the mapping is crude (no frequency band differentiation, no rhythmic tracking). But this is the first time we've had any visible audio-driven behavior in the generated output.

- Investigate remaining response delay even without prev-latent conditioning

- Once coarse correspondence is solid, gradually reintroduce visual diversity
